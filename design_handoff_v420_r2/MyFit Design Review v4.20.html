The system is no longer fighting itself. Resolved

The v3.42 review opened with "the bones are good." v4.20 moves past that — the bones are good and the muscle around them does what it says. Tap targets are honest. Day chips are sticky. The Complete pill floats. Dark mode no longer has rough edges. Removed/archived days have their own quiet container. The Reset button is in an overflow menu where it can't be hit by accident. The toast is at the top, out of the way of the floating bars. None of these are individually flashy; collectively they shift the product from "promising single‑user prototype" to "comfortable daily driver."

That's the right moment to stop reviewing the surface and start asking what features would change the experience next. The rest of this document is the answer.

02
Five high‑leverage feature ideas
Ordered by return on effort. All five fit the single‑file model, none require new data schemas.

#1 · Surface the stalled‑lift nudge as a target, not a label High

The detection logic already exists — v2-nudge watches for stalled lifts (last 2 sessions ≤ prev 2 + 0.5) with high effort (RPE ≥ 8) and flags them with a small inline note. That's a passive nudge that asks the user to do the math.

The same condition could prefill the next set's target: italic ghost 140 lb in the weight field with a small "+5" pill above it. Tap to accept. The detection is 100% built. The surface is what's missing.

  • Stalled at RPE ≥ 8 → no bump (recommend hold, optionally suggest +1 rep)
  • Stalled at RPE ≤ 7 → +2.5–5 lb suggested (machine‑specific rounding from exerciseMeta.equipment)
  • PR last session → carry forward the new weight + the same rep target
  • Untouched for 14+ days → fall back to "last weight" ghost (no nudge)
Recommend

Don't store a "recommended" value. Compute on render from bestEst1RM + exerciseMeta.equipment + RPE trend. Show as italic ghost (same treatment as the existing prefill) with a +X pill that disappears once typed into. No new keys, no sync change.

#2 · The PR moment, when it happens High

The app already computes bestEst1RM and renders a "PR" chip in the Lifts sub‑tab. But the moment a PR is actually set — the rep that beats the previous best estimated 1RM — passes silently. The user types 9 at 145 lb, taps the checkbox, sees nothing different, moves on.

Two pieces fix this for a single line each in logic:

  • Inline pill: when the set the user just typed yields a new best estimated 1RM for that lift, the set tile gets a faint yellow halo and a tiny "PR" badge.
  • Toast at completion: when a session is saved with ≥1 PR set, the top‑center toast says "Session saved · 2 new PRs" instead of the generic "saved" — no new modal, no celebration animation, no confetti.

The point is signal, not spectacle. The thing that makes daily logging stick is noticing that something happened.

Recommend

Compute on input blur, not on save — the user gets the feedback while still standing at the machine. PR detection: new bestEst1RM for this histEx vs. prior sessions only (ignore in‑progress sets so an early set doesn't "PR" against itself). Persist nothing — re‑derive on every render.

#3 · Plate math, inline under the weight field Med

Plate Calc currently lives as a button on the Plan screen that opens a modal. The math is useful, but the path is wrong — for a barbell lift you need the breakdown while typing, not three taps away on a different screen. Personal apps can be opinionated here: every weight on a barbell lift gets a tiny plate strip underneath.

Only render for exercises with equipment === 'barbell' (already in metadata). For machines, dumbbells, cables — render nothing. The strip is decorative + informative; no interaction, no taps.

Recommend

Bar weight from a single setting (45 lb default), plates available from a fixed array. Strip color‑codes the standard plate hex (45 = blue, 25 = green, 10 = yellow, 5 = purple, 2.5 = light). Tap the strip → opens the existing modal for bar / plate set adjustments. The modal stays, but it's the rare deep‑dive, not the default path.

#4 · Rest timer learns per‑exercise pattern Med

The rest timer is a single global duration. But a compound barbell squat doesn't need the same rest as a cable tricep pushdown — the metadata pass (Phase B/B2) already tags every entry with a pattern (push/pull/squat/hinge/lunge/carry/core/cardio/isolation/mobility). That's enough information to set a sensible default per row:

  • squat / hinge / carry — 120s
  • push / pull (compound) — 90s
  • isolation — 60s
  • core / mobility — 30s (or no timer)
  • cardio — no rest timer (it's the activity)

The user can still override; the global preference becomes "long" / "short" / "auto" — auto is the new default.

Recommend

One lookup table, one line at the tap‑checkbox handler. Persist override per linkId in a new key rest_overrides_v1 (mirrored in the synced payload). Display the chosen duration in the rest‑timer pill: "Resting · 1:23 · auto 90s · ⏯".

#5 · The Sunday retrospective card Med

Right now there's no end‑of‑week moment. Sessions get logged, the Lifts tab gets richer, but nothing prompts the user to look back. A single card on the Workout tab — visible Sunday evenings through Monday morning, hidden the rest of the week — would carry the whole week's signal in one read:

  • 3 sessions completed · 14 days streak
  • +2.5 lb on chest press · +1 rep on lat pulldown · 2 new PRs
  • One muscle group flagged: rear delts — last logged 11 days ago
  • Suggested next: Try Day 4 (Upper B) tomorrow — based on plan cycle position

It's a low‑interaction surface — read‑only, dismissable, never the primary action. But it's the moment that makes a habit stick. Strava's Sunday email is the same idea; it doesn't need a notification, just a screen presence at the right time.

Recommend

Render gate: day === sunday || (day === monday && hour < 10), and at least 1 logged session in the previous 7d. Data is all derived (no new keys). Tap to expand → inline list of PRs with set details; second tap → goes to Lifts filtered to that exercise.

04
The More tab is becoming a junk drawer
It made sense when there were 3 things in it. There are now ten, all at the same level.

Ten flat surfaces, no sub‑structure Med

The R1 IA decision (5 tabs → 4 + More) was clearly the right one — but the More tab inherited everything the other four tabs didn't want. As of v4.20 it holds: cloud sync, primary‑device toggle, primary‑toggle hint, pull‑from‑cloud, on‑device snapshots launcher, cloud‑archive launcher, backup‑code / export / import launcher, dark‑mode toggle, "Developer" disclosure (Test mode), Update, version footer. Ten interactive rows, none grouped.

The fix isn't more screens — it's three lightweight section headers + a deliberate order. The data‑risky stuff at the top (where you'd look in a panic), the cosmetic stuff in the middle, the developer / about stuff at the bottom.

Three subgroups, a stable order, two consolidations Low

The right is a 1:1 collapse — no rows lost. The wins:

  • Cloud sync + primary device → one row. The toggle is the on/off, the secondary line carries the primary‑device state. They're already coupled functionally.
  • Snapshots + cloud archive + backup → one row ("Backup & restore"). They're three ways of doing the same thing; let the launcher show all three options once you tap in.
  • Appearance gets two more controls: units (from §03) + day‑label length (currently a parked decision in the handoff). Folding them in here gives "Appearance" something to do.
  • Rest timer as its own row — the auto/long/short setting from §02 #4 lives here.
Recommend

Three <h3 class="more-group"> headers between rows. No new screen, no new disclosure pattern — just visual grouping. The "Developer" disclosure stays exactly as v4.19 shipped it; it just moves to the bottom of its group.

05
The three deferred CSS sweeps · honest cost / benefit
5.2 (type), 5.3 (color/radius/shadow), 5.5 (.btn vocab) — flagged as low payoff in the handoff. Here's the call from a fresh read.

5.2 · Type tokens · partially done is the right place to stop Low

The handoff is right: this is invisible to the user. The v2 card already uses the tokens; the older Progress / Plan / Modal markup hardcodes 11–13px values that the user will never consciously notice. Token discipline matters when multiple people are working on a file — for a single‑user, single‑maintainer build, the cost of the find‑replace pass exceeds the benefit.

Recommend

Skip. Only do this if you're about to touch the older markup for an unrelated reason — opportunistic, not standalone.

~70% of the file off‑tokens but visually consistent enough that the user can't tell. Cost: 1 day of careful regex + smoke. Payoff: 0 visible change.

5.3 · Color / shadow tokens · same answer, with one exception Low

The radius slice landed in v4.20 — that's the part the user can feel (modals now agree on --r-lg). The remaining color & shadow drift is again invisible from outside the file. The one thing worth doing on this front separately: replace the ~6 ad‑hoc grays (#888780 · #B4B2A9 · #C8C6BD · #2C2C2A) with their nearest --muted/--ink/--border token in only the dark‑mode rules, where contrast actually matters and any drift is visible.

Recommend

Targeted dark‑only pass. Light mode is fine as‑is. Roughly 2 hours of work, fixes the half‑dozen places where the 4‑hex‑deep gray ladder actually shows up.

5.5 · .btn vocab consolidation · do this one Med

This is the only deferred sweep with a real user‑visible payoff. There are 5+ "primary affirmative" buttons in the file (.complete-btn, .modal-btn-primary, .mgr-add-btn, .tray-act, .deload-plan-btn, .patch5-rerun-btn), and they don't look like each other. A user clicking the "Save session" modal button and the "Add exercise" mgr button gets two different button shapes for two affirmative actions. Consolidation is a real polish gain.

You'd also get a smaller file (these classes all have their own dark‑mode rule too, so the savings compound).

Recommend

Define .btn + .btn-primary / .btn-ghost / .btn-danger / .btn-sm / .btn-block. Each existing class becomes an alias for one month, then gets deleted. The deload button is the only one that arguably wants its own treatment (a destructive recovery action) — it can keep .deload-plan-btn as a one‑off composition: .btn .btn-danger .btn-block.

06
Three long bets
Bigger, less certain. Worth thinking about — not necessarily worth doing soon.
Long bet · 01

Notes are unsearchable history

Every session can carry per‑exercise notes ("right shoulder twinge", "felt strong"). They're there forever, but there's no way to ask "every session where I noted shoulder pain". Lifts gives you weight history; nothing gives you narrative history.

A search field at the top of Sessions, scanning notes + session‑level note, would unlock the data you've already been writing.

Long bet · 02

Apple Health two‑way

Body Measurements (Weight / Waist / Body Fat) duplicates data the user almost certainly already has in Health. Even a one‑way read (pull Weight from Health on Body panel load) would mean the user stops double‑entering.

PWA can't talk to HealthKit directly, but a one‑shot iOS Shortcut + Files share could fill body_v1 with no schema change.

Long bet · 03

The cycle position is invisible

The plan has a cycle (Day 1 → 2 → 3 → 4 → 5 → repeat). The user manually picks the chip. The app already knows the most‑recently‑logged day and could surface "Up next: Day 3" on Workout with one tap.

Removes one decision per session. Especially valuable on weeks where the user skipped a day and forgot where they were.

07
Risk · the additive paradigm has gravity
Every version of this app has been additive (it's the right call for data‑safety). But "additive only" has a cost — eventually every screen carries dead weight, because removing anything is a re‑baseline.

You're carrying a lot of "this is here because removing it would re‑baseline index.html.bak".

The handoff is honest about this: the byte‑identity gate on verify.js means any change to rendered card markup intentionally fails the smoke test, which is why a clean‑up of EXERCISE_DATA markup is treated as a separate event. That's correct for safety. The cost is that small adjustments to the card surface have become disproportionately expensive — and so the surface accretes.

A separate verify pass that locks data identity (the parsed plan, exercise list, history JSON) without locking markup identity would let the card breathe. It's the difference between protecting the user's logs and protecting the user's view of their logs. Only the first one actually matters.
